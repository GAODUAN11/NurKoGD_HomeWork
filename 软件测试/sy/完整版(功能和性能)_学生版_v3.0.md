# 实验一：自动化功能测试

## 1. 实验目的

1. **掌握现代自动化测试栈**：深入理解基于 Python + Pytest + Playwright 的 UI 自动化测试体系。
2. **实战 POM 设计模式**：通过页面对象模型（Page Object Model）实现业务逻辑与测试脚本的解耦。
3. **理解数据驱动测试（DDT）**：学习使用 YAML 管理测试数据，实现正反向业务场景的精准断言。
4. **构建系统可观测性**：通过异常捕获与网络拦截技术，建立“代码层”与“数据流”双轨审计体系。
5. **掌握可视化分析技术**：学习使用 Allure 框架构建业务实况看板。
6. **模拟 CI/CD 集成**：掌握多环境切换及一键自动化执行脚本的编写。

## 2. 实验环境

- **硬件**：PC 机（推荐内存 8G+）
- **操作系统**：Windows 10/11 或 Linux/macOS
- **编程语言**：Python 3.x
- **核心库**：`pytest`, `playwright`, `allure-pytest`, `pyyaml`
- **浏览器引擎**：Chromium (Playwright 内核)
- **报告工具**：Allure Framework

------

## 3. 实验内容与步骤

以某个Web程序为被测试软件，编写UI自动化测试脚本。

### 3.1 基础版

实现用户（增、删、改、更多、状态开关）的 Playwright 录制脚本

### 3.2 增强版

#### 3.2.1 登录身份复用策略

- **任务**：解决若依系统的验证码登录问题。
- **要求**：实现“一次登录，多次复用”，通过某种机制实现测试身份的自动注入，确保后续业务用例在不显式运行登录动作的情况下即可进入系统。

#### 3.2.2 POM 页面对象建模

- **任务**：实现测试逻辑和页面定位的分离

- **要求**：

  - `pages/base_page.py`: 封装通用底层逻辑，如统一的页面加载 open 方法（处理基础 URL 拼接）

  - `pages/login_page.py`: 负责登录动作及身份凭证的初始化（用于身份复用策略）。

  - `pages/user_page.py`: 封装用户管理模块的原子动作（增、删、改、更多、状态开关）。

  - `pages/components/`: 复杂组件抽离（可选，建议用于表单或侧边栏）。
    - `menu_component.py`: 侧边栏/导航组件，负责模块间跳转逻辑。
    - `user_form_component.py`: 负责新增、修改用户表单内部的交互细节。

#### 3.2.3 **数据驱动测试（DDT）和全场景断言设计**：

- **任务**：实现测试逻辑与数据解耦。
- **要求**：
  - **CRUD 闭环测试**：实现用户管理模块的新增、修改（具体行）、删除（具体行）、以及“更多”（具体行）下拉菜单、“状态”开关（具体行）的原子操作测试。
  - **YAML 动态读取**：`user_data.yaml` 必须包含**场景标识**、**用例描述**、**输入数据**及**预期结果**。
  - **输入数据正反全场景**
    - 输入数据应该包含，正反全场景覆盖，不仅验证“能通过”的正向路径，更要模拟异常输入（格式错误、唯一性错误【账号重复、手机号重复】、逻辑错误【删除系统管理员】）等负向场景.
  - **断言全场景**
    - 所有断言期望值均须映射在 YAML 中并动态读取，严禁在测试脚本中硬编码。
    - 断言必须覆盖正反全场景，断言要具体，不能笼统
      - **正向（成功）场景**：落实“业务应该成功时是否真正成功的双重核验”。
        - **UI 表现层断言**：通过 `expect()` 校验页面 Toast 提示或 DOM 状态的变更。
        - **物理数据层断言**：通过搜索确认记录数量（`.count()`）或关键字段符合预期。
      - **业务负向（失败）场景**：落实“业务失败时是否实现精准拦截的核验”。
        - 通过 `expect()` 校验特定的错误提示、DOM 元素状态（如单显红字报错）或弹窗内容，确保系统按预期逻辑

#### 3.2.4 测试数据自动化管理机制

- **任务**：通过 **Setup阶段**为测试案例准备前置数据，通过 **Teardown 阶段**清理测试脚本产生的数据，防止对数据库的污染。

- **要求**：
- **前置准备 (Setup)**：为测试案例准备数据。脚本必须以 `user_data.yaml` 为数据模板，并在运行期通过代码**注入随机变量**（如：用户名 `test_{timestamp}`）。
  
- **后置清理 (Teardown)**：无论测试执行成功或者失败，在该用例生命周期结束时，产生的测试垃圾数据都必须被精准移除（“谁造的数据，谁删除”，“造了什么数据，就删除什么数据”）。
  
- **技术实现路径**：
  
  - **方案 A：UI （初级）**：通过 UI 点击完成 Setup/Teardown，流程直观但执行速度较慢。
  
  - **方案 B：API （进阶/推荐）**：优先探索基于 `page.request` 的接口调用逻辑。能有效避开 UI 层的随机不稳定因素，极大地提升测试效率。


#### 3.2.5 可视化报告与审计日志

- **任务**：发现失败”，更是“精准诊断失败”。可视化报告 给人看，审计日志是 审计日志是给“自动化工程”看的。

- **要求**：

- 1. **可视化报告 (Allure) —— “业务实况看板”：**
  - **维度管理**：使用 `@allure.feature` 标注功能模块，使用 `@allure.story` 标注具体业务场景。
  - **路径追踪**：在 POM 方法中使用 `@allure.step`；在测试流程内部使用 `with allure.step` 组织逻辑块。
  - **双重证据留存**：
    - **Allure 嵌入**：用例触发 `failed` 或 `error` 时，自动调用 `allure.attach` 将截图嵌入可视化报告，作为“第一现场”直观定界。
    - **物理文件存储**：同步将截图以标准文件形式（如 `{timestamp}_{test_id}.png`）保存至 `logs/screenshots/` 目录。确保在进行大数据量统计分析或离线回溯时，依然持有“物理证据”。

- 2. **审计日志 (Logs) —— “工程诊断”：**
  - **“代码层”诊断 (`failure.log`)**：
    - **记录时机**：当用例触发 `error` 或 `failed` 时，记录“为什么失败”。
    - **格式规范**：必须包含 `test_id` (用例路径), `phase_name` (错误阶段), `exception_type` (错误类型), `message` (错误简述), `traceback` (堆栈详情) 以及 `timestamp`。
      - **`phase_name` (错误阶段)**：必须区分 `setup`、`call`、`teardown`、`未知阶段`。
   - **`exception_type` (错误类型)**：根据异常根因，精细化记录为以下三类：
        - **`TimeoutError` (链路不通)**：如元素定位超时、页面加载超时、接口响应超时等。
        
        - **`AssertionError`（业务逻辑错）**：即 YAML 中的预期结果与实际 UI/数据表现不符。
        
        - **`Runtime_Error` (脚本代码写烂了)**：如语法错误 (SyntaxError)、文件缺失 (FileNotFoundError)、变量引用错误等脚本本身的代码质量问题。
  - **“数据流”记录 (`network_trace.log`)**：采用**网络拦截技术**记录接口交互的“全过程”。
    
    - **为什么要这个“数据流”记录 (`network_trace.log`)？**：对于“代码层”诊断 (`failure.log`)中的 **`TimeoutError` (链路不通)**、 **`AssertionError`（业务逻辑错）**需要进一步诊断
      - 对于**`TimeoutError` (链路不通)**需要进一步诊断：环境问题(网络断了)、测试脚本定位错误（测试脚本错了）
      -  对于**`AssertionError`（业务逻辑错）**需要进一步诊断：是被测系统的前端错误还是被测系统的后端错误
    
    - **记录时机**：拦截全量 API 交互，实现非侵入式全链路审计。
    - **格式规范**：自动提取 **请求信息**（`method`, `url`, `payload`——仅限新增、变更操作）与 **响应状态**（`status`, `response_body`——JSON 结果）以及 `timestamp`。
    


#### 3.2.6 多环境切换与命令行集成 (CI/CD 模拟)

- **任务**：实现一套代码适配多套环境，实现命令行集成。

- **要求**：

  - **多环境切换**

    - **参数驱动**：通过命令行参数（如 `pytest --env=test`）动态加载对应的环境配置文件（如 `config/test.yaml`）。

    - **动态注入**：脚本中的 `Base URL`、`测试账号`等变量必须从配置对象读取，严禁在页面对象（POM）中硬编码环境地址。

- **一键启动脚本 (`run.sh` / `run.bat`)**
  - 脚本需集成“环境检查 -> 执行测试 -> 生成 Allure 报告 -> 自动打开报告”的完整流水线。

# 实验二：自动化性能测试

## 1. 实验目的

1. **掌握性能测试全流程**：学习从脚本开发、场景建模到结果分析的闭环测试流程。
2. **实战接口关联技术**：掌握通过 `JSON Extractor` 提取 Token 并利用全局属性（Property）实现跨线程组身份鉴权的方案。
3. **理解数据驱动测试（DDT）**：利用 CSV 参数化与函数助手（如 `__time`）解决并发下的数据冲突。
4. **构建复杂压测场景**：模拟真实生产环境的“读多写少”混合负载模型。
5. **掌握命令行压测与可视化分析**：通过 CLI 模式执行压测并生成 HTML 性能分析报告。

## 2. 实验环境

- **硬件**：PC 机（推荐 8G 内存以上）
- **操作系统**：Windows 10/11 或 Linux
- **软件工具**：Apache JMeter (v5.x+), JDK 1.8+
- **被测系统**：若依管理系统（前后端分离版）

## 3. 实验内容与步骤

### 3.1基础版-微信公众号测试

#### 3.1.1 脚本开发与组件设计

- **全局鉴权方案**：在 `Setup Thread Group` 中实现登录，将会话 Token 转化为全局属性
- **业务流实现**：编写创建标签、查询所有标签、修改标签类核心取样器。

#### 3.1.2 数据驱动与精准断言

- **参数化处理**：通过 `CSV Data Set Config` 实现用标签的批量创建，同时使用 `__time()` 或 `__RandomString()` 函数生成随机后缀，确保并发新增时数据不会因重复而报错。
- **精准断言 (Response Assertion)**：
  - **协议断言**：校验 HTTP 状态码（如 `200`）及响应耗时，确保通信链路正常。

  - **业务逻辑断言**：
    - **结构校验**：验证 JSON 包含特定业务字段（如 `rows`, `total`），确保接口契约一致。

    - **内容校验**：验证返回数据的逻辑正确性（如：查询结果不为空、新增的数据项在列表中真实存在）。

#### 3.1.3 压测模型建模（混合业务流）

- **分流策略**：使用吞吐量控制器（Throughput Controller）模拟真实比例：
  - 查询（100%）、增改（30%）。
- **执行策略**：配置 50-100 并发线程，设置 Ramp-up 周期，持续运行 3-5 分钟。

### 3.2 增强版-若依测试

#### 3.2.1 脚本开发与组件设计

- **全局鉴权方案**：在 `Setup Thread Group` 中实现**管理员**登录，将会话 Token 转化为全局属性，确保查、增、删、改等请求的鉴权一致性。
- **业务流实现**：编写登录、查询用户列表、详情浏览、新增用户、修改用户等 5 类核心取样器。

#### 3.2.2 数据驱动与精准断言

- **参数化处理**：通过 `CSV Data Set Config` 实现登录、新增用户数据参数化，其中新增用户同时使用 `__time()` 或 `__RandomString()` 函数生成随机后缀，确保并发新增时数据不会因重复而报错（如手机号重复）。

- **精准断言 (Response Assertion)**：
  - **协议断言**：校验 HTTP 状态码（如 `200`）及响应耗时，确保通信链路正常。
  
  - **业务逻辑断言**：
  - **结构校验**：验证 JSON 包含特定业务字段（如 `rows`, `total`），确保接口契约一致。
    
  - **内容校验**：验证返回数据的逻辑正确性（如：查询结果不为空、新增的数据项在列表中真实存在）。

#### 3.2.3 压测模型建模（混合业务流）

- **分流策略**：使用吞吐量控制器（Throughput Controller）模拟真实比例：
  - 查询（100%）、详情浏览（60%）、增改（30%）、登录（10%）。
- **执行策略**：配置 50-100 并发线程，设置 Ramp-up 周期，持续运行 3-5 分钟。
