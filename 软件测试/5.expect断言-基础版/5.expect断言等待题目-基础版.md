## 一、 简答题

1. 请简述 Playwright 中 `expect` 断言的 "Web-First" 特性及其核心优势。

   - **Web-First** 指的是 Playwright 的 `expect` 断言会自动重试（轮询）直到满足条件或超时，而不是立即失败。

   - 核心优势

     ：

     - 自动处理异步操作（如网络请求、动画、延迟渲染等）；
     - 减少手动添加 `wait_for_*` 的代码；
     - 提高测试稳定性与可读性。

2. `expect` **状态断言** 常用API？

   - `expect(locator).to_be_enabled()`：元素启用
   - `expect(locator).to_be_disabled()`：元素禁用
   - `expect(locator).to_be_visible()`：元素可见
   - `expect(locator).to_be_hidden()`：元素不可见（包括 `display: none`）
   - `expect(locator).to_be_attached()`：元素在 DOM 中
   - `expect(locator).to_be_detached()`：元素不在 DOM 中

3. `expect` **内容与属性断言 **** 常用API？

   - `expect(locator).to_have_text("xxx")`：文本内容匹配
   - `expect(locator).to_have_value("xxx")`：表单值匹配
   - `expect(locator).to_have_attribute("attr", "value")`：属性值匹配
   - `expect(locator).to_have_class("class-name")`：包含指定 class
   - `expect(locator).to_have_id("id")`：ID 匹配

4. **`expect` 页面 (Page) 级别断言**常用API？

   - `expect(page).to_have_url("...")`：URL 完全匹配
   - `expect(page).to_have_url(re.compile(r"..."))`：URL 正则匹配
   - `expect(page).to_have_title("...")`：页面标题匹配

5. `expect`  **取反断言 (Not)****常用API？

   - `expect(locator).not_to_be_enabled()`
   - `expect(locator).not_to_be_visible()`
   - `expect(locator).not_to_have_text("xxx")`
   - `expect(page).not_to_have_url("...")`
   - 所有正向断言前加 `.not_` 即可取反

6. 什么情况下需要进行**“负向等待”？**

   - **负向等待**用于验证某个状态**不再存在**，例如加载遮罩消失、错误提示移除等。

   - **代码示例**

     ```python
     expect(page.locator(".loading-spinner")).to_be_hidden()
     # 或
     expect(page.locator(".loading-spinner")).not_to_be_visible()
     ```

7. 什么情况下需要使用**正则表达式断言**？

   - 当期望内容**部分匹配**或**格式固定但值动态**时（如时间戳、ID、URL 路径等）。
   - 示例：验证 URL 包含 `/dashboard`，但后面可能带参数。

   ```python
   expect(page).to_have_url(re.compile(r"/dashboard"))
   ```

8. 什么情况下需要使用**列表/集合断言 ?**

   - 当需要验证多个元素的文本、属性等构成一个**预期集合**时。
   - Playwright 支持通过 `all()` 配合 `expect` 对多个 locator 进行批量断言（需配合 `locator.all()`）。

9. 什么是**软断言 (Soft Assertions) **？什么场景适合用**软断言**？

   - **软断言**：即使某个断言失败，测试仍继续执行后续断言，最后统一报告所有失败。
   - **适用场景**：表单多字段校验、页面多个信息项验证等，避免因一个字段错误中断整个检查流程。
   - Playwright 中通过 `with page.expect_soft_assertions():` 或 pytest 的 `pytest-check` 插件实现（官方推荐使用 `expect(...).soft` 在 v1.40+）。

   > 注：Playwright v1.40+ 支持 `expect(locator).soft.to_have_text(...)` 实现软断言。

### 实战2 ：被禁用的提交 (Enabled State)

**场景：** 用户必须勾选“我已阅读协议”，下方的“注册”按钮才会取消 `disabled` 属性。
**要求：**

1. 验证“注册”按钮初始为禁用状态。
2. 勾选协议复选框。
3. 验证按钮变为可用状态并点击。

### 实战3：负向等待场景

**场景：** 点击“生成报告”按钮后，页面会出现一个 `.loading-spinner`，等待约 3 秒后，加载器消失，按钮变为“下载报告”状态且背景色包含 `active` 类。
**要求：**

1. 验证加载器 `.loading-spinner` 消失。
2. 验证按钮文本变为“下载报告”。
3. 验证按钮拥有 `active` 这个 CSS 类（使用正则匹配）。

### 实战4： 骨架屏+数据异步加载场景

**场景：** 进入商品详情页，用户先看到一个灰色的方块（骨架屏），3 秒后后端数据返回，灰色小方块变成真正的“立即购买”按钮，且会注入 `data-id` 属性。
**要求：**
1. 断言按钮起初包含 `skeleton` 类。
2. 断言按钮最终文本变为“立即购买”。
3. 断言按钮拥有属性 `data-id` 且值为 `ITEM_999`

### 实战5：表单字段批量检查（软断言）

**场景：** 个人中心页面有五个信息展示项：用户名、手机号、邮箱、等级、积分。
**要求：**

1. 使用**软断言**检查这五个字段的初始值。
2. 即使“手机号”校验失败，脚本也必须继续检查“邮箱”和“等级”。
